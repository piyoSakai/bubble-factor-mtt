# MTT Bubble Factor / ICM ツール（ローカル WebApp）企画書

## 1. 概要・目的

### 1.1 背景

- Android 向けに「MTT の **Bubble Factor（バブルファクター）** を含む ICM 系ツール」が Google Play 上では実用的に乏しい。
- iOS の **Tournament Cruncher（ICM）** クラスができることを **ベースライン** とし、可能なら **より多くのシチュエーション**（払い戻し人数、PKO、将来拡張）をカバーする **ローカル完結型 WebApp** を開発する。

### 1.2 プロダクトゴール

| ゴール | 内容 |
|--------|------|
| 機能 | iOS 主要アプリの方向性を踏まえつつ、**MVP は BF / RP / ICM / ChipChop に絞る** |
| 拡張 | 同一モデル内で扱える範囲を **可能な限り広げる**（後述のフェーズ分け） |
| 計算 | **サーバーにスタック・賞金情報を送らない**（オフライン利用可能） |
| UX | **スマホ利用を主戦場**と想定し、操作・可読性・待ち時間の体験を最優先。人数増でも **フリーズや無応答を避ける**（非同期・上限・近似の明示） |

---

## 2. 最小入力モデル（コア）

ICM と Bubble Factor の **本質的な入力**は次の 2 点で足りる（ブラインドレベルや M/Q は **ICM 上は不要**）。

| 入力 | 内容 |
|------|------|
| **各プレイヤーのスタック** | **チップ実数** または **BB 換算**のどちらか一方で統一。内部では同一単位の非負整数（または有理数）として扱い、合計チップがプライズ配分の前提と整合すること。 |
| **各順位の賞金** | 残り人数分の払い戻し（1 位〜k 位）。通貨は任意だが、全プレイヤーの $EV と同一スケールにする。 |

この前提だけで **現在の $EV（ICM）** と、仕様で固定したオールイン後配分に基づく **Bubble Factor** が計算できる。

**補足（M / Q / BB）**: ハリントンの **M** や平均対比の **Q** は、ライブ感覚の目安にはなるが **ICM の定義式には入らない**。**BB 数**も「スタックの別表現」に過ぎず、**BB で入力するなら BB をチップ単位として揃えればよい**だけである。よって本プロダクトの **MVP コアには含めない**（必要なら将来「表示オプション」として検討）。

## 3. ベンチマーク（iOS「Tournament Cruncher」相当）

以下を **機能パリティ候補** とする（名称は製品側で調整可）。**太字**は上記最小入力で届く範囲。  
ただし **MVP は BF / RP / ICM / ChipChop に集中**し、spot tool 的な拡張は後続フェーズに回す。

1. **ICM 計算**（スタック一覧＋払い戻し表から各プレイヤーの $EV）
2. **ChipChop**（チョップ配分の別モデル）
3. **ファイナルテーブル等のチョップ比較**（ICM deal vs ChipChop 等）
4. **Bubble Factor 計算**（プレイヤー間の ICM 圧力の指標）
5. **プッシュ / コール等の簡易 ICM 意思決定支援**（チップ移動を仮定した $EV 比較）
6. M ゾーン / Q 比など **ハリントン系の補助指標**（**コア外**。実装する場合はブラインド・アンティ・人数を追加入力とする）

参考: 既存アプリは **最大約 10 名・払い戻し 10 段** などの上限を掲げている例がある。本プロダクトは **上限を引き上げつつ、重い場合は計算モードで吸収**する方針とする。

## 4. 理論・アルゴリズム

### 4.1 ICM（Independent Chip Model）

- **入力**: 全プレイヤーのチップ量、残りプライズ構造（1 位〜k 位）、正規化された賞金配列（合計 100% または同一通貨）。
- **出力**: 各プレイヤー i の期待賞金 \(EV_i\)（$EV）。
- **標準実装**: **Malmuth–Harville（MH）** 型の再帰／動的計画法による近似が広く用いられる。
  - 厳密に全順位を列挙する素朴実装はプレイヤー数増に伴い **計算爆発**しやすい（Wikipedia 等でも「多人数では計算困難」とされる）。
- **本プロダクトの方針**:
  - **フェーズ A**: MH ベースの **正確（実装定義上の厳密）な ICM** を、**プレイヤー数 N と払い戻し段数 P に応じた上限内**で提供する（ビットマスク DP 等の最適化を検討）。
  - **フェーズ B**: 大人数 MTT 向けに **Monte Carlo サンプリング** または公開文献・実装可能な **高速近似 ICM** を追加し、**誤差と推奨用途を UI に明示**する。

### 4.2 Bubble Factor（定義の確認）

出典: ICMIZER 公式ブログ（*What is the Bubble Factor in Poker and Why Should I Use It*）における説明。

プレイヤー A が、特定の相手 B との **オールイン（チップが A から見て「負けた場合／勝った場合」でスタックが定まる）** を想定するとき:

- \(EV_{\mathrm{begin}}\): 手前の ICM 期待値（賞金プールに対する割合または金額で統一）
- \(EV_{\mathrm{lose}}\): **負けた後**のスタック分布での A の ICM 期待値
- \(EV_{\mathrm{win}}\): **勝った後**のスタック分布での A の ICM 期待値

**Bubble Factor（A 対 B、当該オールイン状況）**:

\[
\mathrm{BF}(A \rightarrow B)
= \frac{EV_{\mathrm{begin}} - EV_{\mathrm{lose}}}{EV_{\mathrm{win}} - EV_{\mathrm{begin}}}
= \frac{\Delta_{\mathrm{lose}}}{\Delta_{\mathrm{win}}}
\]

- 分母が 0 に近い（勝っても $EV がほぼ増えない）場合は **定義上アウトプット不可** とし、UI で「不適用・数値不安定」と表示する。
- **MVP の割り切り**: BF は **「対称オールインの study tool」** として定義する。すなわち、**blind / ante / 既存 pot / サイド pot / 複数オールイン / 引き分け**は **MVP では扱わない**。
- **MVP のチップ移動ルール**: 各オーダーペア \((i, j)\) について「**行プレイヤーが列プレイヤーのオールインに直面してコール／フォールドを判断する**」想定で、**有効スタック = \(\min(\text{stack}_i, \text{stack}_j)\)** を全額ぶつける **対称オールイン**を仮定する。勝ち時は行プレイヤーがその有効スタック分を獲得し、負け時は同量を失う。チップ移動ルールは **全セルで 1 通りに固定**する。
- **全対全マトリクス**: 上記固定ルールで \(\mathrm{BF}(i \rightarrow j)\) を算出する。計算回数は **\(O(N^2)\) 回の ICM 評価**に比例するため、**Web Worker でバッチ化**し、人数上限超過時は行単位の遅延計算や近似モードとする（§6.2）。

**簡易コール閾値（近似）**（同ブログ）:

\[
\text{必要エクイティ（近似)} \approx \frac{\mathrm{BF}}{\mathrm{BF} + 1}
\]

- 実戦では antes・サイドポット・複数オールイン・将来のブラインド上昇（FGS）等でズレるため、**「目安」ラベルと注記**を付ける。

### 4.3 Bubble Factor マトリクス（UI・見方）

**目的**: GTO Wizard の *Bubble Factors* タブに相当する **ヒートマップ付き正方行列**を提供し、テーブル全体の ICM 圧力の見取り図にする。

**軸ラベル**: ポジション名（UTG 等）は **使わない**。ユーザーが入力した **Player1, Player2, …**（または任意の表示名）を **行・列の両方**に同一順序で並べる。

**セルの意味**（[GTO Wizard — What is the Bubble Factor in poker tournaments?](https://blog.gtowizard.com/what-is-the-bubble-factor-in-poker-tournaments/) に準拠）:

| 要素 | 説明 |
|------|------|
| **行（縦軸）** | **意思決定者** … 相手のオールインに **直面してコール判断をする側** |
| **列（横軸）** | **オポーネント** … **オールインしてきた側**（「列のプレイヤーがシューブした」と読む） |
| **セル主値** | **Bubble Factor**（行が列に対しての BF）。数値が **大きいほどタイトにプレイすべき**（ICM 圧力が強い） |
| **対角** | 自分自身との組み合わせは **空欄／非表示**（意味がないため） |
| **色（ヒートマップ）** | BF の大小で着色。**赤系＝高 BF・非常にタイト**、**緑系＝低 BF・Chip EV に近い**、という **GTO Wizard の視覚慣習**を踏襲（閾値は実装時に調整可） |

**コールに必要なエクイティ（$EV ベースの目安）**（同記事・ICMIZER と同型）:

\[
\text{Required Equity} \approx \frac{\mathrm{BF}}{\mathrm{BF} + 1}
\]

**Risk Premium（リスクプレミアム）**（GTO Wizard の説明）:  
「**Chip EV でコールに必要なエクイティ**に、**何パーセントポイント加算すればよいか**」を表す。**BF と同じ結論の別表現**。UI は **上段に Risk Premium（例: +4.1%）**、**下段に BF** の **GTO Wizard 型の二段表示**を MVP で採用する。

**MVP における Chip EV の簡易仮定（Wizard 互換の目安）**:  
ポットオッズを毎回入力させず、**「等額オールインのコール側・Chip EV 損益分岐が 50% エクイティのスポット」** と固定する（記事の数値例 [BF=1.18 → Required≈54% → RP=+4.1%](https://blog.gtowizard.com/what-is-the-bubble-factor-in-poker-tournaments/) と整合）。

\[
E_{\$} = \frac{\mathrm{BF}}{\mathrm{BF} + 1}, \quad
\text{Risk Premium (\%pt)} = 100 \times (E_{\$} - E_{\mathrm{chip}})
\]

ここで **\(E_{\mathrm{chip}} = 0.5\)**（50%）を **MVP の既定**とする。  
※ 実ポットでは Chip EV 必要エクイティはオッズ依存のため、**「対称オールイン・50% 基準の上乗せ」**であることを **常時ラベル／ツールチップ**で明示する。v1.x で **ポット比率入力**により \(E_{\mathrm{chip}}\) を上書き可能にしてもよい。

**PKO 注記**: GTO Wizard の記事どおり、バウンティがあると **BF が 1 未満**になり得る（**マイナスの Risk Premium**＝Chip EV よりワイドにコールしうる）。標準 MTT（バウンティなし）を MVP の主対象とし、PKO は §4.5 の拡張で扱う。

### 4.4 ChipChop

- チョップ交渉で用いる **チップ量比例配分** と、ICM 配分の **並列表示・差分** を提供する（式は標準的な ChipChop 定義に従う）。

### 4.5 拡張モデル（優先度付き）

| 優先度 | 内容 | 備考 |
|--------|------|------|
| 高 | 標準 ICM + BF マトリクス + **Risk Premium（50% Chip EV 仮定）** + チョップ | MVP コア |
| 中 | **spot tool 化**（blind / ante / pot size / pot odds 入力） | BF / RP を実戦スポット寄りに拡張 |
| 中 | **PKO / プログレッシブバウンティ** の $EV 近似 | モデル定義を仕様書で固定（賞金＋バウンティ期待） |
| 中 | 同一 ICM 上での **複数オールイン（カテゴリカル）** の粗いシナリオ | 組み合わせ爆発に注意 |
| 低 | **FGS（Future Game Simulation）** | ICM の再帰拡張。計算コスト大・別モジュール推奨 |

## 5. 機能要件（要件一覧）

### 5.1 必須（MVP）

- **スタック**: 全員分。単位は **チップ** または **BB** のいずれかに **統一**（UI でモード選択し、内部は同一スケールに正規化）
- **払い戻し**: 各順位の賞金（残り人数に対応する行数）
- プレイヤー名（任意）
- **ICM $EV** の表・グラフ（簡易）
- **Bubble Factor**（仕様で固定したオールイン・チップ移動ルールに基づく）
- **Bubble Factor マトリクス**（§4.3）: 全プレイヤーを行列入力順で軸に取り、各セルに **Risk Premium（%pt）と BF の二段表示**、ヒートマップ。対角は空欄。Risk Premium は **§4.3 の Chip EV = 50% 簡易仮定**。**色だけに情報を載せない**（数値は常に見える／読める）
- **スマホ最適化**（§6.4）: 主要画面・マトリクスを **モバイルファースト**で設計（**UX は最優先要件**）
- **ChipChop** と **ICM** の比較（チョップ用途を重視する場合は MVP に含める。最小入力のまま利用可能）
- シナリオの **保存 / 読込**（LocalStorage またはファイルエクスポート）
- **計算はブラウザ内のみ**（ネットワーク送信なし）

### 5.2 あると良い（v1.x）

- **ミッドランジ／バブル帯の簡易 BF・ICM モード**（ファイナルテーブル想定以外）  
  **狙い**: 残り **約 20% ～ ITM 直後** など、**全順位の払い戻しを細かく入れきれない局面**でも、**おおよその ICM 圧と BF の目安**を得る。可能ならパラメータで **もっと広い残り人数帯**にも拡張。  
  **入力例（案）**:
  - **賞金のおおよその傾斜**（例: トップヘビー／フラット／サテライト風のプリセット、または上位 % の粗いスライダー）
  - **ITM ライン**（払い戻し人数、または「残り何人中何位まで金銭圏」）
  - **フィールド残り人数**（`# of players left`）
  - **自分のテーブルの状況**（着席人数分のスタック。テーブル外は下記の仮定で合成）
  **モデル上の仮定（要 UI 明示）**: テーブル外の `(残り人数 − テーブル人数)` を **均等チップ**、または **ショート帯／ミドル帯の 2～3 バケツ** など **少数の仮想スタック**にまとめ、**圧縮した賞金ベクトル**（金銭圏手前数席＋代表ジャンプのみ等）で MH-ICM。結果は **推定・学習用**であり、FT 厳密モードと **数値を並べて誤解しない**ラベルを付ける。
  **UI ラベル方針**: 「気になる人がちゃんと気付ける」ことを優先し、**画面内に常設の `推定値` バッジ／注記**を置く。少なくとも **結果見出し付近**、**共有・保存時**、**詳細説明（ツールチップ or 情報パネル）** の 3 か所で推定モードであることを認知できるようにする。
- **Chip EV 基準の上書き**（ポット比率から \(E_{\mathrm{chip}}\) を入力 → Risk Premium を精緻化）
- CSV / JSON インポート
- 大人数時、**ヒーロー行のみ**先に計算して残りをバックグラウンド（計算量削減）
- プリセット払い戻し（代表的 MTT / SNG）
- ダークモード・**色以外の冗長コーディング**（アクセシビリティ）

### 5.3 研究寄り（v2 以降）

- PKO
- 大人数向け高速 ICM モードとの **誤差表示**
- FGS（深度・ブラインドスケジュール入力）

## 6. 非機能要件

### 6.1 ローカル計算・プライバシー

- スタック・賞金・名前は **既定で外部に送信しない**。
- 利用解析・CDN 経由のスクリプトは **最小化**し、オフライン配布（PWA / 単一 HTML バンドル）も選択肢に入れる。

### 6.2 パフォーマンス・UX

| 項目 | 方針 |
|------|------|
| メインスレッド | **ブロックしない**（Web Worker または WASM ワーカー） |
| 入力中 | **デバウンス**（例: 300ms）後に再計算キュー |
| 大規模 | **「正確モード」上限**（例: N≤15, P≤15 を初期案、実測で調整）を超えたら **近似モード** または **計算を分割**。BF マトリクスは **\(N(N-1)\) セル**分のシナリオ ICM なので **特に Worker・チャンク処理** |
| 長時間 | **プログレス・キャンセル**（AbortController） |
| 結果の一貫性 | 同一入力では **再現可能**（乱数シード固定オプション for MC） |

### 6.3 品質

- 既知の小規模ケースに対する **スナップショットテスト**（ICM 合計＝賞金総額、ChipChop 合計一致等）
- TypeScript 厳格モード、`any` 回避

### 6.4 スマホ最適化・UX（MVP で必須）

**方針**: 利用シーンは **Android / iPhone のブラウザ**を主と想定し、デスクトップは拡張表示。**UX は機能と同列の成功基準**とする。

| 項目 | 要件の例 |
|------|----------|
| レイアウト | **モバイルファースト**のレスポンシブ。主要操作は **親指届く範囲**（下部に主要 CTA を置く等）を検討 |
| タッチ | タップ領域 **最低 44×44 CSS px 相当**、誤タップ防止の余白 |
| BF マトリクス | **横スクロール＋固定ヘッダ／固定先頭列**、または **セルタップで詳細（RP・BF）をボトムシート表示**。ヒートマップは補助で、**RP・BF の数値が主**（§5.1） |
| 入力 | `inputmode="decimal"`、数値キーボード誘導、ステップ入力の取りこぼし防止 |
| 可読性 | フォントサイズ・コントラスト・ダークモードは **少なくとも OS の「文字サイズ／減少モーション」に過度に反発しない** |
| 待ち時間 | 計算中は **スケルトン／プログレス**、**キャンセル可能**。長い処理は **チャンク**し UI を止めない（§6.2） |
| PWA | **オプションだが推奨**（ホーム追加、起動しやすさ）。オフラインは計算ロジックと合わせて検討 |
| 安全領域 | `viewport-fit=cover` 利用時は **ノッチ／ホームインジケータ**と被らない余白 |

受け入れの目安: **幅 390px 前後のビューポート**で主要フロー（入力 → 結果 → マトリクス閲覧）が **ズームなしで完結**すること。

## 7. 技術スタック案

| 領域 | 候補 | 理由 |
|------|------|------|
| UI | React + Vite（または Svelte） | コンポーネント分割・状態管理が容易 |
| 計算 | TypeScript コア + **Web Worker** | UI と分離、メインスレッド保護 |
| 数値 | `number`（64bit float）＋必要なら誤差注記 | 実装コストと互換性のバランス |
| 配布 | PWA + `vite build` 静的ホスト / ローカル `file://` 注意書き | オフライン利用 |

※ `file://` では Worker に制限がある場合があるため、**ローカルは `npm run dev` または簡易 HTTP サーバ**を推奨と明記する。

## 9. リスクと対策

| リスク | 対策 |
|--------|------|
| 多人数 ICM が重い | 上限・Worker・チャンク処理・近似モード。必要なら BF マトリクスは **ヒーロー行優先** または **表示解像度の段階的更新** |
| Bubble Factor の解釈がユーザーによって異なる | MVP は **対称オールインの study tool** と明記し、オールイン後のスタック配分ルールを **図解＋固定仕様** で明示 |
| Risk Premium を GTO Wizard の厳密値と誤認される | **`50% Chip EV 基準`** を常設表示し、ツールチップとヘルプで「簡易仮定」を繰り返し明示 |
| PKO・FGS の「正解」がモデル依存 | 出典・近似であることを UI に表示、設定可能パラメータを限定 |
| ギャンブル規制・ストア審査 | 本件は **研究用計算ツール** としての表示（実装は Web で審査対象外になりやすい） |
